# 前端架构规范

## 新增需求

### 需求：模块化项目结构

系统前端必须采用模块化的目录结构，按职责分层组织代码。

#### 场景：目录结构符合分层架构

- **当** 开发者查看前端项目结构时
- **那么** 应看到以下目录组织：
  - `src/api/`：API 接口层，按业务模块拆分文件
  - `src/components/`：组件层，包含通用组件和布局组件
  - `src/views/`：页面视图层，对应路由页面
  - `src/store/`：状态管理层，使用 Pinia Stores
  - `src/utils/`：工具层，包含通用工具函数
  - `src/router/`：路由配置层

#### 场景：API 接口按业务模块拆分

- **当** 开发者需要调用特定业务模块的接口时
- **那么** 应能在对应的 API 文件中找到接口定义
- **并且** API 文件应按以下模块划分：
  - `api/auth.js`：认证接口（登录、登出、同步角色）
  - `api/student.js`：学生管理接口
  - `api/coach.js`：教练管理接口
  - `api/order.js`：订单管理接口
  - `api/goods.js`：商品管理接口
  - `api/account.js`：账号管理接口
  - `api/rbac.js`：权限管理接口

#### 场景：组件按类型分类

- **当** 开发者需要使用或修改组件时
- **那么** 应能在对应的组件目录中找到组件定义
- **并且** 组件应按以下类型分类：
  - `components/common/`：可复用的通用 UI 组件（Modal、Table、Pagination、Loading）
  - `components/layout/`：布局组件（Header、Sidebar、MainContent）
  - `views/`：完整的业务页面组件（Login、StudentManagement 等）

### 需求：构建工具和技术栈

系统前端必须使用现代化的构建工具和技术栈。

#### 场景：使用 Vite 作为构建工具

- **当** 开发者启动开发服务器时
- **那么** 应使用 Vite Dev Server
- **并且** 应支持热模块替换（HMR）
- **并且** 应配置代理，将 `/api` 请求转发到后端服务器（`localhost:5001`）

#### 场景：使用 Vue 3 技术栈

- **当** 开发者编写前端代码时
- **那么** 应使用 Vue 3 框架
- **并且** 应使用 Composition API 或 Options API（保持一致性）
- **并且** 应使用 Vue Router 4 进行路由管理
- **并且** 应使用 Pinia 进行状态管理
- **并且** 应使用 Axios 进行 HTTP 请求

#### 场景：生产环境构建

- **当** 执行生产构建命令时
- **那么** 应使用 Vite 构建工具生成优化后的静态文件
- **并且** 构建产物应输出到指定的目录（如 `dist/` 或 `frontend-dist/`）
- **并且** 构建产物应包含压缩和优化的 JavaScript、CSS 文件
- **并且** 后端服务器应能提供这些静态文件

### 需求：HTTP 请求封装

系统前端必须统一封装 HTTP 请求，实现请求拦截和错误处理。

#### 场景：自动添加 JWT Token

- **当** 前端发起 HTTP 请求时
- **那么** 应自动从 `localStorage` 读取 `token`
- **并且** 应在请求头中添加 `Authorization: Bearer {token}`

#### 场景：处理 401 未授权错误

- **当** 后端返回 401 状态码时
- **那么** 应自动清除本地存储的 `token`
- **并且** 应重定向到登录页面

#### 场景：统一错误处理

- **当** HTTP 请求失败时
- **那么** 应返回格式化的错误信息
- **并且** 应支持在调用处进行自定义错误处理

### 需求：状态管理

系统前端必须使用 Pinia 进行全局状态管理。

#### 场景：认证状态管理

- **当** 用户登录成功时
- **那么** 应在认证 Store 中保存以下状态：
  - `isLoggedIn`：是否已登录
  - `username`：用户名
  - `isSuperAdmin`：是否为超级管理员
  - `token`：JWT Token（存储在 `localStorage`）

#### 场景：权限状态管理

- **当** 用户登录或切换角色时
- **那么** 应在权限 Store 中保存以下状态：
  - `enabledPermissions`：启用的权限 action_id 列表
  - `menuTree`：动态菜单树数据

#### 场景：状态持久化

- **当** 用户刷新页面时
- **那么** 应从 `localStorage` 恢复 `token`
- **并且** 应根据 `token` 判断登录状态
- **并且** 如果 `token` 有效，应自动恢复用户信息

### 需求：路由管理

系统前端必须使用 Vue Router 进行页面路由管理。

#### 场景：路由配置

- **当** 开发者配置路由时
- **那么** 应定义以下路由：
  - `/`：登录页（Login.vue）
  - `/home`：主应用页面，包含布局和嵌套路由
  - `/home/students`：学生管理页
  - `/home/coaches`：教练管理页
  - `/home/orders`：订单管理页
  - `/home/goods`：商品管理页
  - `/home/accounts`：账号管理页
  - `/home/rbac`：权限管理页

#### 场景：路由守卫

- **当** 用户访问需要认证的页面时
- **那么** 应检查 `localStorage` 中是否存在有效的 `token`
- **并且** 如果不存在 `token`，应重定向到登录页
- **并且** 如果存在 `token`，应允许访问

#### 场景：路由模式

- **当** 配置路由时
- **那么** 应使用 Hash 模式（`createWebHashHistory`）
- **原因**：避免需要后端配置 fallback 路由

### 需求：组件复用

系统前端必须提供可复用的公共组件。

#### 场景：模态框组件

- **当** 页面需要显示弹窗时
- **那么** 应使用通用的 `Modal.vue` 组件
- **并且** 该组件应支持以下功能：
  - 自定义标题
  - 自定义内容（通过插槽）
  - 确认和取消按钮
  - 打开/关闭动画

#### 场景：表格组件

- **当** 页面需要展示数据列表时
- **那么** 应使用通用的 `Table.vue` 组件
- **并且** 该组件应支持以下功能：
  - 自定义列配置
  - 数据绑定
  - 行操作按钮（编辑、删除等）

#### 场景：分页组件

- **当** 数据列表需要分页时
- **那么** 应使用通用的 `Pagination.vue` 组件
- **并且** 该组件应支持以下功能：
  - 当前页码显示
  - 总页数计算
  - 上一页/下一页按钮
  - 页码跳转

#### 场景：加载动画组件

- **当** 页面正在加载数据时
- **那么** 应使用通用的 `Loading.vue` 组件
- **并且** 该组件应显示加载动画和提示文本
- **并且** 该组件应支持全屏遮罩和局部遮罩两种模式

### 需求：样式管理

系统前端必须合理组织样式代码。

#### 场景：全局样式

- **当** 应用启动时
- **那么** 应加载全局样式文件 `public/styles.css`
- **并且** 全局样式应保持与原项目一致

#### 场景：组件样式

- **当** 开发单文件组件（.vue）时
- **那么** 应在 `<style scoped>` 标签中编写组件样式
- **并且** 应避免样式冲突
- **并且** 可以复用全局样式类

#### 场景：CSS 变量

- **当** 需要定义主题色、间距等公共样式值时
- **那么** 应在全局样式中定义 CSS 变量
- **并且** 在组件中使用 CSS 变量引用这些值

### 需求：后端接口兼容性

系统前端必须与现有后端 API 完全兼容。

#### 场景：API 接口路径不变

- **当** 前端调用后端接口时
- **那么** 应使用与原项目相同的接口路径
- **例如**：
  - 登录：`POST /api/login`
  - 获取学生列表：`GET /api/students`
  - 新增学生：`POST /api/students`

#### 场景：请求格式不变

- **当** 前端发送请求时
- **那么** 请求头、请求体格式应与原项目一致
- **并且** 应使用相同的参数名和数据类型

#### 场景：响应处理不变

- **当** 前端接收响应时
- **那么** 应按照原项目的响应格式解析数据
- **并且** 应正确处理 `code`、`message`、`data` 字段
- **例如**：
  ```javascript
  {
    "code": 0,
    "message": "success",
    "data": { ... }
  }
  ```

#### 场景：认证流程不变

- **当** 用户登录时
- **那么** 应使用 JWT Token 认证机制
- **并且** Token 应存储在 `localStorage` 中
- **并且** 每次请求应自动携带 Token
- **并且** 401 错误应触发登出并重定向到登录页

### 需求：功能等价性

系统前端重构后必须提供与原版完全一致的功能。

#### 场景：学生管理功能

- **当** 用户访问学生管理页面时
- **那么** 应能执行以下操作：
  - 查看学生列表
  - 按 ID、姓名、年级、状态筛选学生
  - 新增学生
  - 编辑学生
  - 删除学生
  - 分页浏览学生列表

#### 场景：教练管理功能

- **当** 用户访问教练管理页面时
- **那么** 应能执行以下操作：
  - 查看教练列表
  - 按 ID、姓名、性别、科目、状态筛选教练
  - 新增教练
  - 编辑教练
  - 删除教练
  - 分页浏览教练列表

#### 场景：订单管理功能

- **当** 用户访问订单管理页面时
- **那么** 应能执行以下操作：
  - 查看订单列表
  - 按订单 ID、用户 ID、状态筛选订单
  - 新增订单
  - 编辑订单
  - 查看子订单
  - 管理订单状态

#### 场景：商品管理功能

- **当** 用户访问商品管理页面时
- **那么** 应能执行以下操作：
  - 查看商品列表
  - 管理品牌、分类、属性
  - 新增商品
  - 编辑商品
  - 删除商品
  - 管理商品状态

#### 场景：账号管理功能

- **当** 用户访问账号管理页面时
- **那么** 应能执行以下操作：
  - 查看账号列表
  - 按 ID、手机号、角色筛选账号
  - 新增账号
  - 编辑账号
  - 分配角色
  - 管理账号状态

#### 场景：权限管理功能

- **当** 用户访问权限管理页面时
- **那么** 应能执行以下操作：
  - 查看角色列表
  - 管理角色权限
  - 查看权限列表
  - 查看菜单管理
  - 配置角色权限关系

#### 场景：权限控制

- **当** 用户没有某项操作权限时
- **那么** 应隐藏对应的操作按钮
- **并且** 应基于 `enabledPermissions` 列表进行权限检查
- **并且** 超级管理员应拥有所有权限

### 需求：开发环境配置

系统前端必须提供良好的开发环境配置。

#### 场景：开发服务器启动

- **当** 开发者执行 `npm run dev` 命令时
- **那么** 应启动 Vite 开发服务器
- **并且** 应在默认端口（如 5173）运行
- **并且** 应自动打开浏览器

#### 场景：API 代理配置

- **当** 开发服务器运行时
- **那么** 应将所有 `/api` 开头的请求代理到后端服务器（`localhost:5001`）
- **并且** 应支持跨域请求

#### 场景：热模块替换

- **当** 开发者修改代码时
- **那么** 应自动触发热模块替换（HMR）
- **并且** 页面应快速更新而不刷新整个页面

### 需求：生产环境部署

系统前端必须支持生产环境部署。

#### 场景：构建命令

- **当** 开发者执行 `npm run build` 命令时
- **那么** 应使用 Vite 构建生产版本
- **并且** 构建产物应输出到指定目录
- **并且** JavaScript 和 CSS 应被压缩和优化

#### 场景：静态文件服务

- **当** 后端服务器启动时
- **那么** 应提供前端静态文件服务
- **并且** 应将前端构建产物目录配置为静态文件根目录
- **并且** 访问根路径时应返回 `index.html`

#### 场景：生产环境测试

- **当** 部署到生产环境前
- **那么** 应在本地测试生产构建版本
- **并且** 应验证所有功能正常
- **并且** 应验证 API 调用正确

### 需求：代码质量和规范

系统前端必须遵循代码质量和规范标准。

#### 场景：代码格式化

- **当** 开发者编写代码时
- **那么** 应遵循一致的代码格式
- **并且** 应配置 ESLint 进行代码检查
- **并且** 应使用 Prettier 进行代码格式化（可选）

#### 场景：文件命名规范

- **当** 创建新文件时
- **那么** 应遵循以下命名规范：
  - 组件文件：PascalCase（如 `StudentManagement.vue`）
  - JS 文件：camelCase（如 `request.js`、`auth.js`）
  - 常量文件：UPPER_CASE（如有需要）

#### 场景：代码注释

- **当** 编写复杂逻辑时
- **那么** 应添加必要的注释说明
- **并且** API 接口函数应包含 JSDoc 注释
- **并且** 公共组件应包含使用说明

### 需求：文档和说明

系统前端必须提供完善的文档。

#### 场景：项目 README

- **当** 开发者首次查看项目时
- **那么** 应提供 `frontend/README.md` 文档
- **并且** 文档应包含以下内容：
  - 项目结构说明
  - 技术栈介绍
  - 安装和启动步骤
  - 构建和部署步骤
  - 开发规范

#### 场景：API 文档

- **当** 开发者需要调用 API 时
- **那么** 应提供 API 接口文档
- **并且** 文档应说明每个接口的路径、方法、参数、响应格式

#### 场景：组件文档

- **当** 开发者需要使用公共组件时
- **那么** 应在组件文件中提供使用说明
- **或者** 应提供独立的组件文档
